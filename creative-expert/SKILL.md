## 一、阶段3：Hook盲测（Thompson Sampling）

### 执行规则

1. 从 `core-knowledge.md` §一的8类Hook中，基于当前品类的Beta先验采样θ值
2. 若该品类无历史数据，使用Beta(1,1)无信息先验
3. 若该品类非消费品（如技术/SaaS），Hook权重独立维护，不污染消费品先验
4. 输出3个选项：A（θ最高）、B（θ次高）、C（随机探索）

### 输出格式

```
## 【阶段3：Hook盲测】

| 臂 | Hook类型 | 文案 |
|:---|:---|:---|
| A | [类型] | "[文案]" |
| B | [类型] | "[文案]" |
| C | [类型] | "[文案]" |

> 请选择A、B或C。不选择则默认采用臂A。

**默认选择: 臂A**（如用户未在规定时间内选择）
```

## 二、阶段4：15秒脚本生成

### 执行规则

1. 基于选定的Hook（用户选择或默认选择），生成标准格式脚本
2. 格式必须严格遵循：`镜头编号 | 景别·运镜·时长 | 画面描述 | 口播`
3. 必须包含复播彩蛋（镜头4最后1.5秒的开放循环）

### 输出格式

```
## 【阶段4：15秒脚本】

**文案**: [≤5字，情感锚点]

| 镜头 | 时间 | 景别·运镜 | 画面描述 | 口播 |
|:---|:---|:---|:---|:---|
| 1 | 0-3s | [景别]·[运镜] | [含声音Hook+视觉冲突] | "[口播]" |
| 2 | 3-8s | [景别]·[运镜] | [核心卖点视觉化] | "[口播]" |
| 3 | 8-12s | [景别]·[运镜] | [使用场景+结果对比] | "[口播]" |
| 4 | 12-15s | [景别]·[运镜] | [CTA+复播彩蛋] | "[口播]" |
```

## 三、阶段5：贝叶斯诊断（强制执行）

### 执行规则

1. **必须执行**。即使无用户反馈，也使用16个失败案例的Beta先验
2. 致命层任一项不通过 → 标注"禁止提交"，但**仍继续输出阶段6的修复版**
3. 核心层/卫生层问题 → 记录并在阶段6输出修正版

### 诊断检查清单

```
致命层:
□ 视觉冲突前3秒是否出现？
□ 声音Hook存在？
□ AIGC合规三大禁区无触碰？

核心层:
□ 垂直信号前5秒口述+字幕双出现？
□ 原生感：60%+真实材质占比？
□ 复播彩蛋是否存在？

卫生层:
□ 无"不/别/没"负向锚点词？
□ 单段≤100词？
□ 口播标注/停顿标记？
□ 可验证细节替代抽象形容词？
```

### 输出格式

```
## 【阶段5：贝叶斯诊断】

| 层级 | 状态 | 问题 |
|:---|:---|:---|
| 致命层 | ✅ / ❌ | [如有问题，描述] |
| 核心层 | ✅ / ⚠️ | [如有问题，描述] |
| 卫生层 | ✅ / ⚠️ | [如有问题，描述+自动修正方案] |

**贝叶斯概率**:
- P(Viral): [值]
- 95% CI (Bootstrap n=1000): [下限, 上限]
- 决策建议: [建议]
- 校准状态: [基于先验 / 基于用户反馈 / 基于投放数据]
```

## 五、🆕 阶段6.5：内部自评迭代

### 执行规则

1. **阶段6输出完成后自动触发**。不需要编排者额外指令。
2. 重新对已输出的4段Prompt执行三层诊断
3. 发现问题 → 在当前输出中给出修正后的Prompt
4. 最多迭代3轮，第3轮后强制输出最终版本

### 自评维度

```
致命层自评:
□ 视觉冲突是否仍然存在？
□ 声音Hook是否仍然存在？

□ 口播是否标注/停顿标记？
□ 可验证细节是否已替换抽象词？
```

## 六、🆕 自检熔断机制 (Self-Correction & Circuit Breaker)

> **这是本SKILL最关键的机制。你必须在输出最终答案前，强制执行以下自检流程。**

### 6.1 完整性检查 (Completion Check)

在输出全部内容后，你必须逐一核对以下条目。如果任何一项缺失，**立即补充**。

- □ 是否已输出**阶段3**的3个Hook选项？
- □ 是否已输出**阶段4**的标准15秒4镜头脚本？
- □ 是否已输出**阶段5**的贝叶斯诊断摘要（含P(Viral) + 95% CI）？
- □ 是否已输出**阶段6**的4段Seedance 2.0原子化提示词？
- □ 是否已输出**阶段6.5**的内部自评结果（含修正和收敛判定）？

### 6.2 熔断规则 (Circuit Breaker Rules)

| 触发条件 | 动作 |
|:---|:---|
| 致命层不通过 | 🛑 立即标注"致命层拦截"，但仍输出修复版Prompt（而非完全拒绝输出） |
| 核心层≥2项不通过 | ⚠️ 输出警告 + 修复版Prompt，标注修正项 |
| 卫生层问题 | 🔧 自动修正 + 输出修正版Prompt |
| 自评3轮后仍未收敛 | 🔄 输出当前版本 + 标注剩余问题清单 |
| 输出不完整（缺失任何阶段） | 🔄 **立即补充缺失阶段的内容**，不得以"需要更多信息"为由跳过 |

### 6.3 完整性检查输出格式

在**全部内容输出完毕后**，你必须附加以下检查清单：

```
---
【自检熔断 · 完整性确认】
· 阶段3 (Hook盲测): ✅ 已输出 / ❌ 缺失 → [立即补充]
· 阶段4 (脚本生成): ✅ 已输出 / ❌ 缺失 → [立即补充]
· 阶段5 (贝叶斯诊断): ✅ 已输出 / ❌ 缺失 → [立即补充]
· 阶段6 (原子化提示词): ✅ 已输出 / ❌ 缺失 → [立即补充]
· 阶段6.5 (内部自评): ✅ 已输出 / ❌ 缺失 → [立即补充]
· 熔断状态: [正常 / 已触发 → 原因]

```

## 八、全局铁律

1. **闭环强制**: 必须完整输出阶段3→4→5→6→6.5，禁止跳过任何阶段
2. **即使无用户反馈**: 阶段5也必须输出（使用Beta先验，标注"基于先验|未校准"）

4. **即使自评未收敛**: 最多3轮后必须输出最终版本，不可无限循环
