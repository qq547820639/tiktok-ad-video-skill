## 三、核心问题：算术不自洽

报告中有多处数值不遵循自己的公式。这是最需要修的问题，因为它直接违背了 v2.0/v3.0 最重要的设计目标——数学诚实性。

### 问题 1：U_domain 计算错误

报告给出：
```
U_creative = 1.000, U_product = 1.000, U_execution = 0.894, U_delivery = 1.000
U_domain = 0.963
```

按公式验算：
```
U_domain = 0.35 × 1.000 + 0.25 × 1.000 + 0.25 × 0.894 + 0.15 × 1.000
         = 0.350 + 0.250 + 0.224 + 0.150
         = 0.974
```

**报告 0.963，正确值 0.974。** 差 0.011。

### 问题 2：RiskLoad 计算错误

报告列出各分项值：
```
TextRisk = 1.00, EditRisk = 0.60, ClaimRisk = 0.30, ProductFidRisk = 0.30, CTARisk = 0.60
RiskLoad = 1.90
```

按公式验算：
```
RiskLoad = 0.30 × 1.00 + 0.20 × 0.60 + 0.20 × 0.30 + 0.15 × 0.30 + 0.15 × 0.60
         = 0.300 + 0.120 + 0.060 + 0.045 + 0.090
         = 0.615
```

**报告 1.90，正确值 0.615。** 差 3 倍。

### 问题 3：R_risk 与 RiskLoad 不自洽

报告说 `R_risk = 0.954, RiskLoad = 1.90`。

反推：`exp(-0.25 × 1.90) = exp(-0.475) ≈ 0.622`，不是 0.954。

如果 RiskLoad 正确值为 0.615：`exp(-0.25 × 0.615) = exp(-0.154) ≈ 0.857`。

**报告 0.954，正确值 ≈0.857。** 无论哪个 RiskLoad 都对不上。

### 问题 4：StructureScore 最终值不自洽

报告给出的计算过程：
```
100 × 0.963 × 1.000 × 0.954 × 1.056 - 10.0 = 86.4
```

验算（用报告自己的值）：
```
100 × 0.963 × 1.000 × 0.954 × 1.056
= 100 × 0.963 × 0.954 × 1.056
= 100 × 0.963 × 1.0074
= 100 × 0.9701
= 97.01
97.01 - 10.0 = 87.0
```

**报告 86.4，按报告自己的中间值算得 87.0。** 差 0.6。

如果用正确的中间值：
```
100 × 0.974 × 1.000 × 0.857 × 1.056 - 10.0
= 100 × 0.974 × 0.857 × 1.056 - 10.0
= 100 × 0.974 × 0.905 - 10.0
= 100 × 0.881 - 10.0
= 88.1 - 10.0
= 78.1
```

**如果严格按公式和正确的 N 值计算，SS ≈ 78.1，不是 86.4。** 差约 8 分。

### 问题 5：CF 计算错误

报告给出 `CF = 0.82`。

验算：
```
0.93^0.20 ≈ 0.986
0.85^0.15 ≈ 0.976
1.00^0.15 = 1.000
1.00^0.15 = 1.000
0.90^0.20 ≈ 0.979
0.75^0.15 ≈ 0.958

CF = 0.986 × 0.976 × 1.000 × 1.000 × 0.979 × 0.958
   = 0.986 × 0.976 = 0.962
   × 0.979 = 0.942
   × 0.958 = 0.903
```

**报告 0.82，正确值 ≈0.90。** 差 0.08。

### 问题 6：HeuristicUncertainty 严重偏差

报告给出 σ 分项：±3, ±2, ±1, ±1, ±5, ±5，合成后 ±4。

验算：
```
sqrt(3² + 2² + 1² + 1² + 5² + 5²)
= sqrt(9 + 4 + 1 + 1 + 25 + 25)
= sqrt(65)
≈ 8.06
```

**报告 ±4，正确值 ±8.1。** 差一倍。报告说"四舍五入"，但 sqrt(65) 四舍五入是 8，不是 4。

### 问题 7：PostReadyScore 计算错误

报告给出 P1=5, P2=5, P3=4, P4=5, P5=5, P6=4，PostReadyScore=91.0。

**报告 91.0，正确值 94.0。** 差 3.0。

## 五、根因分析

这个问题的根因不是规则设计问题，而是 **LLM 在长上下文中做多步算术的固有弱点**：

1. 报告格式非常长（约 3000+ 字），中间计算分散在多个段落
2. 指数函数（exp）、幂函数（^0.20）、几何均值（sqrt）等非线性计算容易出错
3. 多个中间值需要传递到后续计算，任何一步出错会传播
4. 模型倾向于给出"看起来合理"的数值，而不是严格按公式计算

## 六、修复建议

### 方案 A：分步计算 + 校验层（推荐）

在状态机中增加一个 **Score Verification 状态**：

```
[状态 3.2.5] 算术校验
  → 逐项重算 U_domain, G_gate, R_risk, B_bonus, P_penalty
  → 计算 SS = 100 × U × G × R × B - P
  → 逐项重算 CF
  → 逐项重算 σ 合成
  → 逐项重算 NoPostGate
  → 逐项重算 PostReadyScore
  → 若任何值与初始计算偏差 > 0.02，以重算值为准
```

### 方案 B：简化公式，减少非线性计算步骤

当前公式链太长，每一步都是误差源。建议简化：

```
简化前（v3.0）:
q_i → U_creative(sqrt) → U_product(sqrt) → U_execution(sqrt) → U_delivery
→ U_domain(加权和) → G_gate(5项乘) → R_risk(exp) → B_bonus(exp)
→ P_penalty(5项和) → SS(clamp) → CF(6项幂乘) → σ(6项sqrt)

简化后:
q_i → U_domain(加权和，不用几何均值) → G_gate(3项核心乘) 
→ SS = 100 × U × G × min(1, B) - P
→ σ = 查表合成（不用 sqrt）
```

### 方案 C：输出时附带验算行

```
【验算】
U_domain = 0.35×1.000 + 0.25×1.000 + 0.25×0.894 + 0.15×1.000 = 0.974
R_risk = exp(-0.25 × 0.615) = exp(-0.154) = 0.857
SS = 100 × 0.974 × 1.000 × 0.857 × 1.056 - 10.0 = 78.1
```

这样用户可以直接看到计算过程并验证。
